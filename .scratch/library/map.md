# Wayfinder Map: Library

Map for the effort to adapt upstream Shelfmark into a per-user ebook library builder. Personal fork of `calibrain/shelfmark` — library management is explicitly a non-goal upstream, so this work stays in the fork and does not get PR'd back.

## Destination

A Shelfmark fork where each user has a per-user library of books (metadata-provider-backed works), backed by persistent routes (`/library`, `/library/:bookId`) and a book/detail UX similar to Grimmory. Adding a book to the library is the spine of the experience: it routes the user to the release-list page if no files exist globally, or to the new book detail page if files already exist. The book detail page surfaces per-format downloads, dedup indicators across users, and one-click Send-to-Kindle. Downloads never auto-add to library; library entries are explicit.

## Notes

- **Domain**: ebook library management layered on top of Shelfmark's existing search/download pipeline. Not a Calibre replacement; a thin library UX on top of the download machine.
- **Tracker**: local-markdown under `.scratch/library/`. Map = this file. Tickets = `.scratch/library/issues/NN-<slug>.md`.
- **Skills every session should consult**: `/domain-modeling` for any ticket touching the book/library data model; `/grilling` for any design/UX decision ticket; `/implement` when picking up an implementation-shaped task. The codebase has no ORM and uses hand-rolled `ALTER TABLE` migrations (`shelfmark/core/user_db.py:196-271`) — schema tickets must produce migrations in that style.
- **Personal fork policy**: no upstream PRs. Code follows existing repo conventions (Flask + raw sqlite3 on the back; Vite + React 19 + Tailwind 4 on the front; `make checks` is the gate). Where upstream's Project Scope section in `readme.md:213-224` directly conflicts with our work, amend `readme.md` in the fork — that's a deliberate divergence, not a regression.
- **Standing preferences**: no new external deps without a ticket weighing them. Reuse the existing email-output SMTP machinery (`shelfmark/download/outputs/email.py`) for Send-to-Kindle — do not introduce a new mailer. The codebase has no `books` / `authors` / `sources` tables — those are net-new schema. **Frontend router correction (via #07)**: `react-router-dom ^7.18.1` is already installed and `BrowserRouter` wraps `<App>` at `src/frontend/src/main.tsx:17`; `<App>` renders `<Routes>` with `/login` + `/*` at `App.tsx:2853-2875`. Library routes are **not** the first real routes — the login route is. The decision is how to introduce nested routing inside the authenticated app shell (`mainAppContent`), not whether to adopt a router.
- **Git tracking strategy**: `.scratch/library/` is canonical tracker state, not code — it lands on `main` directly, always, separate from any feature/prototype branch the code work rides on. Rule for any session editing the map or an issue: commit+push the `.scratch/` churn to `main` before resuming code work (or at minimum before claiming/resolving a ticket, so the claim is visible to other sessions). Code stays on its feature branch as today; the tracker never rides a branch and is never entangled with a code PR's merge fate. Single-user posture — no cross-session merge ceremony assumed.

- **One grab → multiple files — graduated via #13.** Resolved: `download_history` is schema (b) one row = one file sharing a non-unique `task_id`; `transfer_file_to_library`/`transfer_directory_to_library` return `list[Path]`; new `task.library_paths` + new `finalize_download_files` delete-sentinel + insert-N-rows; `record_download` retry is explicit DELETE+INSERT. Implementation owned by **#14**. Cleared.
- **Per-file unlink within a grab — graduated via #13 to Out of scope.** Ruled out: releases are unlinked atomically across all sibling file rows sharing `task_id` (decision 4-a-strict). `user_downloads` owns releases, not individual files. Cleared.
- **Admin "download on behalf of user X" linking — graduated via #04.** Under the new `user_downloads` model, an admin queuing on behalf of user X may need to insert both `user_downloads(admin, R)` AND `user_downloads(user_x, R)`. Out of scope for MVP. Surfaces if the admin support case needs it.
- **Non-admin metadata refresh — graduated via #04 (to Out of scope).** See Out of scope below — no refresh mechanism exists for this effort at all.
- **One grab → multiple files — graduated via #13 (see Decisions so far).** Cleared.
- **Per-file unlink within a grab — ruled out of scope via #13.** Cleared.

## Out of scope

- **Upstream PRs.** Library management is explicitly a non-goal upstream (`readme.md:213-224` and `readme.md:230`). This effort produces no upstream changes.
- **Background monitoring of authors, series, or new releases.** Same as upstream's stated non-goals. Author browse (#10) is metadata-provider-on-demand only — no background watchers, no "new release from this author" notifications.
- **Format conversion on demand** (e.g. Calibre `ebook-convert`). The architectural survey confirmed no conversion today. If a book only has mobi on disk, Send-to-Kindle stays greyed; we never transcode.
- **Calibre / Calibre-Web / Audiobookshelf integration.** Shelfmark's existing Library Link feature (a header link) is untouched; we don't push to nor pull from those tools as part of this effort.
- **Metadata refresh (manual or scheduled) — ruled out via #04.** No refresh button on book-detail; no `POST /api/library/books/:book_id/refresh-metadata`; no scheduled refresh for this entire effort. The `books` row written at Add time is final. A future "scheduled refresh like Sonarr/Radarr" lands as a fresh effort, not as a graduation from this map.
- **Admin mutation of other users' libraries — ruled out via #04.** Admin gets instance-wide read for support; cannot remove another user's library row, cannot push to another user's Kindle, cannot unlink another user's release. Admin support workflows that need mutation use direct DB surgery, same as today. Resurfaces only if a "support mode" API is later demanded.
- **Per-user file storage on disk — ruled out via #04.** Files stay global per-instance per `CONTEXT.md`. Per-user storage would multiply disk N× (the qbittorrent/usenet `find_existing` dedup doesn't save you — `_handle_completed_file` still copies into each user's `DESTINATION`), break #02's in-flight race-prevention, and force a fresh settings/permissions/storage layer. ADR `docs/adr/0002-library-file-visibility.md` captures this rejection.
