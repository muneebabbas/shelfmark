# Wayfinder Map: Library

Map for the effort to adapt upstream Shelfmark into a per-user ebook library builder. Personal fork of `calibrain/shelfmark` — library management is explicitly a non-goal upstream, so this work stays in the fork and does not get PR'd back.

## Destination

A Shelfmark fork where each user has a per-user library of books (metadata-provider-backed works), backed by persistent routes (`/library`, `/library/:bookId`) and a book/detail UX similar to Grimmory. Adding a book to the library is the spine of the experience: it routes the user to the release-list page if no files exist globally, or to the new book detail page if files already exist. The book detail page surfaces per-format downloads, dedup indicators across users, and one-click Send-to-Kindle. Downloads never auto-add to library; library entries are explicit.

## Notes

- **Domain**: ebook library management layered on top of Shelfmark's existing search/download pipeline. Not a Calibre replacement; a thin library UX on top of the download machine.
- **Tracker**: local-markdown under `.scratch/library/`. Map = this file. Tickets = `.scratch/library/issues/NN-<slug>.md`.
- **Skills every session should consult**: `/domain-modeling` for any ticket touching the book/library data model; `/grilling` for any design/UX decision ticket; `/implement` when picking up an implementation-shaped task. The codebase has no ORM and uses hand-rolled `ALTER TABLE` migrations (`shelfmark/core/user_db.py:196-271`) — schema tickets must produce migrations in that style.
- **Personal fork policy**: no upstream PRs. Code follows existing repo conventions (Flask + raw sqlite3 on the back; Vite + React 19 + Tailwind 4 on the front; `make checks` is the gate). Where upstream's Project Scope section in `readme.md:213-224` directly conflicts with our work, amend `readme.md` in the fork — that's a deliberate divergence, not a regression.
- **Standing preferences**: no new external deps without a ticket weighing them. Reuse the existing email-output SMTP machinery (`shelfmark/download/outputs/email.py`) for Send-to-Kindle — do not introduce a new mailer. The codebase has no `books` / `authors` / `sources` tables — those are net-new schema. **Frontend router correction (via #07)**: `react-router-dom ^7.18.1` is already installed and `BrowserRouter` wraps `<App>` at `src/frontend/src/main.tsx:17`; `<App>` renders `<Routes>` with `/login` + `/*` at `App.tsx:2853-2875`. Library routes are **not** the first real routes — the login route is. The decision is how to introduce nested routing inside the authenticated app shell (`mainAppContent`), not whether to adopt a router.

## Decisions so far

- [Model the book / library / file relationship](issues/01-book-library-data-model.md) — books are denormalized snapshots keyed on UNIQUE(metadata_provider, provider_book_id) both NOT NULL; user_library(user_id, book_id, added_at) PK pair CASCADE; download_history gets nullable book_id + idx_download_history_book_id; authors are a column not a table (table parked in fog for #12); metadata_json stores the complete raw payload; updated_at bumps on metadata refresh only. ADR `docs/adr/0001-books-as-denormalized-snapshot.md`; glossary in `CONTEXT.md`.
- [Settle the "add to library" flow and the files-exist branch](issues/02-add-to-library-flow-contract.md) — Add always navigates to `/library/:bookId`; no separate release-list route (overrides #07's `/library/:bookId/find`). Add repurposes the existing green `Get`/`+` button (Universal mode only); already-in-library state is "In Library" disabled-button that navigates to `/library/:bookId` Sonarr/Radarr-style. Find Releases is a modal on book detail (re-uses `ReleaseModal`), auto-opened only when `files_exist_globally == false`, gated by env var `LIBRARY_AUTO_FIND_RELEASES` (default `true`). "Files on disk globally" predicate includes in-flight downloads (`final_status IN ('active', 'complete')`) to prevent duplicate-release race; trusts `download_path IS NOT NULL`, defers fs stat to serve-time. Wishlist semantics: file-less library entries are first-class. **#04 amendment:** the in-flight predicate stays valid for auto-open behavior only; the manual Find Releases button is ungated for all users (Q3 grab gate repealed — any user can grab another release anytime, files-exist or not).
- [Design the library backend API surface](issues/04-library-api-contract.md) — routes live in `shelfmark/core/library_routes.py` via `register_library_routes(app, ...)` (matches `activity_routes`/`request_routes`/`admin_routes`/`self_user_routes` `register_*_routes` pattern, NOT a Flask Blueprint — none exists in the codebase). Admin gets instance-wide read of every user's Library + every user's file lists; mutations (Add/Remove/Send-to-Kindle) scoped to admin's own library. Shared global file pool: API returns `formats_on_disk` union across all `download_history` rows for the Book, no per-user attribution. File serving gates on `user_library` Book-membership (NOT `download_history.user_id`) — closes the existing zero-ownership leak in `orchestrator.get_book_data` (`orchestrator.py:342-360`, `queue.py:117-120`) structurally. Any user can grab another release anytime — Q3 grab gate repealed; #02 sub-decision 6 in-flight predicate stays valid for auto-open only, manual Find Releases ungated for all users. New `user_downloads(user_id, history_id)` PK CASCADE link table for per-user-per-release linking — orthogonal to #01's Book-level `user_library`. Unlinking = hard DELETE of the link, file/`download_history` row untouched. Re-add is `INSERT OR IGNORE`. `download_history.user_id` becomes an audit field; `user_downloads.user_id` is the load-bearing column. Orphan concept introduced (download_history row with zero user_downloads entries, not in-flight) — admin cleanup deferred. No pagination for MVP; `?q=` fuzzy via plain `LIKE` on `books.title`/`author` (case-insensitive, ordered by `user_library.added_at DESC`). Add response shape `{book_id, files_exist_globally, in_flight_globally, in_my_library: true}` (idempotent — same payload whether newly inserted or already existed; "in-library" button state driven by separate `GET /api/library/books/:book_id` lookup); 503 on provider down, no `books` row inserted on failure. Send-to-Kindle: new per-user `KINDLE_EMAIL` setting (`user_overridable=True`, visible in `SelfSettingsModal` `delivery` section — the non-admin settings modal + `/api/users/me` endpoint already exist end-to-end); required per-user, no `EMAIL_RECIPIENT` fallback; admin `EMAIL_FROM` required at instance level (env-overrides-UI per existing convention). Send-to-Kindle endpoint `POST /api/library/books/:book_id/send-to-kindle` follows existing error conventions — no invented codes (403/404/400/500 via `_OPERATIONAL_ERRORS`). New `send_file_to_email(file_path, recipient, *, label, subject)` function in `shelfmark/download/outputs/email.py` reuses `compose_email_message` + `send_email_message` + `_get_email_settings()` — no synthetic `DownloadTask`, no new mailer. Metadata refresh explicitly OUT OF SCOPE for this effort (no manual button, no scheduled refresh) — `books` row at Add time is final. ADR `docs/adr/0002-library-file-visibility.md`; ADR 0001 amended (refresh sentence removed); glossary updated with **Orphan** term and File entry amended for `user_downloads`.
- [Implement the books / authors / user_library schema and migration](issues/03-schema-migration.md) — `books`, `user_library`, `user_downloads` created via `_CREATE_TABLES_SQL` (`user_db.py:122-162`); `download_history.book_id` + `idx_download_history_book_id` added by `_migrate_download_history_book_id` (`user_db.py:288-303`) in the existing conditional `PRAGMA table_info` migration idiom, registered in `UserDB.initialize()` at `:247`. **No backfill** — the library starts from a fresh DB; legacy `download_history` rows (if any) keep `book_id IS NULL` and stay invisible to library queries per #01's invariant. - [Frontend routing philosophy](issues/07-frontend-routing-contract.md) — nested `<Routes>` inside `mainAppContent` (`App.tsx:2375`): `/` = `<HomePage>` (search UI extracted mechanically by #08), `/library` = `<LibraryPage>` (#10), `/library/:bookId` = `<BookDetailPage>` (#08), `*` → `<Navigate to="/" replace>`. `:bookId` is the integer `books.id` PK (e.g. `/library/42`), matching #04's API path uniformly across providers. Auth gate stays outer-only — the existing `<Navigate to=/login?return_to=...>` at `App.tsx:2845` already preserves deep-link targets; no per-route guard. Single generic `LibraryNotFound` page for both API 403 (not in your library) and 404 (book id doesn't exist); no Add-to-Library CTA on the error state. - [Send-to-Kindle format priority](issues/05-send-to-kindle-priority-override.md) — default priority `["epub"]` only (the sole Kindle-compatible format shelfmark ships; Amazon deprecated azw3/mobi for Send-to-Kindle 2022). ## Not yet specified

- **Format-picker on existing release-list page.** You want "an option of choosing the format" for downloads. Architectural survey confirms there's no format picker today — the format rides on whatever release the user clicked. Where does the format picker live: on each release row (choose format at download-queue time, source permitting), or only on book detail's per-format buttons? May resolve as a side-effect of the book-detail per-format-download ticket (#08) but stays fog until then.
- **Removing files / re-fetching a release.** You said "we will think about deleting existing files and grabbing another release later maybe as an admin-only action" — admin-only file management is fog; it's not on the route to the destination but is likely to land as a follow-up once the book detail page exists. Resurfaces if the per-format downloads ticket (#08) needs it. **#04 update:** the "re-fetch another release" half of this fog is now resolved (per-user re-grab is allowed; manual Find Releases ungated; release-level metadata is first-class on book-detail; unlink = hard DELETE of `user_downloads` row, file/`download_history` row untouched). The "removing files" half (admin file deletion on disk) stays fog and is now framed as the Orphan cleanup ticket below.
- **Library entry expiry / file-less wishlist cleanup — graduated via #02.** File-less library entries are first-class (wishlist semantics, per #01). Bookshelf (#10) shows a "Find this book" affordance for entries with no files on disk; book-detail empty state (#08) shows the same persistent "Find Releases" button. No expiry, no auto-cleanup — entries stay until the user removes them.
- **Merging books across metadata providers.** Uniqueness today is `(provider, provider_book_id)` — the same work added via Hardcover then Open Library lives as two book rows. Cross-provider merge (Hardcover work ID ↔ Open Library OL*W crosswalk) is fog; not on the route to a working library. Resurfaces after the first user notices duplicates and grumbles.
- **Instance-local authors table for #12's author route.** #01 settled `books.author TEXT` (primary only) and explicitly *did not* create an `authors` table — author identity is provider-specific (Hardcover numeric ids, Open Library `/authors/OL*` keys, Google Books none). If #12 (Author browse MVP) needs an instance-local, provider-agnostic `/authors/:authorId` route rather than provider-scoped `/authors/hardcover/:authorId`, it introduces the table then. Resurfaces when #12 is worked.
- **Library sharing between users.** Per-user library, book entity shared — but can a user *see* another user's library, or only their own? Implicit answer is "own only" but no decision yet. Resurfaces if anyone asks. **#04 update:** admin gets instance-wide read for support (per Q2); mutation stays scoped to admin's own. Ordinary-user-to-ordinary-user visibility stays fog.
- **Direct-mode search results and the library — settled via #02.** Direct-mode (`SEARCH_MODE=direct`) results have no `books` row, no Add-to-Library button, no library entry. Direct-mode keeps its existing direct-download UX unchanged. Worth a sentence in the readme divergence note when the readme-amend ticket lands; not otherwise on the route.
- **Orphan cleanup admin task — graduated via #04.** Files (`download_history` rows) with zero `user_downloads` entries across all users AND not in-flight are Orphans (term now in `CONTEXT.md`). The cleanup mechanism — schedule, retention window, what deletes (file only vs file + row), admin UI surface — is undefined. Lands as a fresh ticket when an admin needs the lever, likely after #08 ships.
- **Pagination + structured filtering for `GET /api/library/books` — graduated via #04.** Deferred to post-MVP per scope-narrowing decision. #04 settled the unpaginated + `?q=` LIKE contract for MVP. Pagination (`?page=`/`?page_size=` or cursor) and structured filters (`?author=`, `?has_files=`) resurface when a user feels the absence at scale (10+ books per user, addressed in a later ticket).
- **FTS5 for fuzzy search — graduated via #04.** Plain `LIKE` on `books.title`/`books.author` chosen for MVP. FTS5 resurfaces if `LIKE` proves too dumb for short-string title/author lookups (apostrophe handling, typos).
- **Unlink mid-flight — graduated via #04.** What happens when a user unlinks an in-flight `final_status='active'` `download_history` row from their `user_downloads`. Backend permits it (hard DELETE on the link; queue keeps running; eventually `download_path` populates and the user can re-link later via `INSERT OR IGNORE`). The UX — disallow with error, or allow with warning — is parked as a decision for #08.
- **Admin "download on behalf of user X" linking — graduated via #04.** Under the new `user_downloads` model, an admin queuing on behalf of user X may need to insert both `user_downloads(admin, R)` AND `user_downloads(user_x, R)`. Out of scope for MVP. Surfaces if the admin support case needs it.
- **Non-admin metadata refresh — graduated via #04 (to Out of scope).** See Out of scope below — no refresh mechanism exists for this effort at all.

## Out of scope

- **Upstream PRs.** Library management is explicitly a non-goal upstream (`readme.md:213-224` and `readme.md:230`). This effort produces no upstream changes.
- **Background monitoring of authors, series, or new releases.** Same as upstream's stated non-goals. Author browse (#10) is metadata-provider-on-demand only — no background watchers, no "new release from this author" notifications.
- **Format conversion on demand** (e.g. Calibre `ebook-convert`). The architectural survey confirmed no conversion today. If a book only has mobi on disk, Send-to-Kindle stays greyed; we never transcode.
- **Calibre / Calibre-Web / Audiobookshelf integration.** Shelfmark's existing Library Link feature (a header link) is untouched; we don't push to nor pull from those tools as part of this effort.
- **Metadata refresh (manual or scheduled) — ruled out via #04.** No refresh button on book-detail; no `POST /api/library/books/:book_id/refresh-metadata`; no scheduled refresh for this entire effort. The `books` row written at Add time is final. A future "scheduled refresh like Sonarr/Radarr" lands as a fresh effort, not as a graduation from this map.
- **Admin mutation of other users' libraries — ruled out via #04.** Admin gets instance-wide read for support; cannot remove another user's library row, cannot push to another user's Kindle, cannot unlink another user's release. Admin support workflows that need mutation use direct DB surgery, same as today. Resurfaces only if a "support mode" API is later demanded.
- **Per-user file storage on disk — ruled out via #04.** Files stay global per-instance per `CONTEXT.md`. Per-user storage would multiply disk N× (the qbittorrent/usenet `find_existing` dedup doesn't save you — `_handle_completed_file` still copies into each user's `DESTINATION`), break #02's in-flight race-prevention, and force a fresh settings/permissions/storage layer. ADR `docs/adr/0002-library-file-visibility.md` captures this rejection.
